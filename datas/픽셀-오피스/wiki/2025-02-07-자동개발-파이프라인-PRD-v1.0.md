## 1. 제품 개요

### 1.1 제품명
**픽셀 오피스 (Pixel Office)** — 자동 개발 파이프라인

### 1.2 한 줄 정의
> 사용자가 피드백이나 요구사항을 입력하면, AI 직원들이 자동으로 기획·디자인·개발·테스트·배포까지 수행하여 앱을 업데이트하는 macOS 앱

### 1.3 비전
누구나 자신이 만들고 싶은 앱을 **아이디어만으로** 구현하고 유지보수할 수 있게 한다.

### 1.4 타겟 사용자
**1인 창업자** — MVP를 빠르게 만들어야 하고, 22개 이상의 수익 앱을 동시에 관리해야 하는 사람

### 1.5 성공 기준
- 22개의 수익 앱을 큰 노력 없이 관리/유지
- 요구사항 입력 → 앱 업데이트까지 사람의 개입 최소화
- 사용자가 코드를 직접 작성할 필요 없음

## 2. 핵심 가치 제안 (Value Proposition)

| 기존 방식 | 픽셀 오피스 |
|-----------|------------|
| 직접 코딩하거나 외주 의뢰 | 요구사항 한 줄 입력 |
| 기획서 → 디자인 → 개발 → 테스트 수동 관리 | AI 직원이 전체 프로세스 자동 수행 |
| 앱마다 별도 관리 | 22개 앱 통합 대시보드 |
| 버그 발생 시 직접 디버깅 | 자동 감지 → 자동 수정 루프 |
| 배포 시 수동 빌드·서명·업로드 | 자동 빌드·배포 파이프라인 |

### 경쟁사 대비 차별점
> **코드 생성이 아닌 개발 프로세스 전체 관리**
- Cursor, Copilot: 코드 편집기 수준
- Replit Agent, Devin: 코드 생성 중심
- **픽셀 오피스**: 기획 → 디자인 → 개발 → QA → 배포 → 유지보수 전체 자동화

## 3. 핵심 기능 요구사항

### 3.1 Phase 1: 입력 시스템 (Input)

#### FR-001: 요구사항 입력 인터페이스
- **설명**: 사용자가 자연어로 요구사항/피드백을 입력하는 메인 진입점
- **상세 요구사항**:
  - 텍스트 입력 (자연어, 한 줄~여러 줄)
  - 스크린샷/이미지 첨부 (UI 관련 피드백 시)
  - 대상 프로젝트 선택 (22개 앱 중 택 1 또는 전체)
  - 우선순위 지정 (긴급/일반/나중에)
- **MVP 범위**: 텍스트 입력 + 프로젝트 선택만 우선 구현
- **수용 기준**: 사용자가 3초 이내에 요구사항 입력을 시작할 수 있어야 함

#### FR-002: 프로젝트 컨텍스트 자동 수집
- **설명**: 요구사항을 해석하기 위해 해당 프로젝트의 현재 상태를 자동 수집
- **상세 요구사항**:
  - 프로젝트 코드베이스 구조 파악
  - 기존 기능 목록 및 의존성 파악
  - 최근 변경 이력 참조
  - 사용 중인 기술 스택 정보 로딩
- **MVP 범위**: 프로젝트 루트의 파일 구조 + 최근 커밋 히스토리 수집
- **수용 기준**: 10초 이내에 프로젝트 컨텍스트 로딩 완료

---

### 3.2 Phase 2: 해석 시스템 (Interpretation)

#### FR-003: 자연어 → 구조화된 작업 분해
- **설명**: 사용자의 자연어 입력을 실행 가능한 작업 단위로 분해
- **상세 요구사항**:
  - 요구사항의 의도 파악 (신규 기능 / 버그 수정 / UI 변경 / 성능 개선 등)
  - 작업을 부서별(기획/디자인/개발/QA)로 분류
  - 각 작업에 필요한 구체적 행동(Action) 목록 생성
  - 예상 소요 시간 및 복잡도 산출
- **MVP 범위**: 기본적인 의도 분류 + 개발 작업 분해
- **수용 기준**: "로그인 버튼 색상 파란색으로 바꿔줘" → 디자인(색상 변경) + 개발(SwiftUI Color 수정) + QA(UI 확인)으로 분해

#### FR-004: 작업 간 의존성 및 순서 결정
- **설명**: 분해된 작업들 간의 의존성을 파악하고 최적 실행 순서를 결정
- **상세 요구사항**:
  - 선행/후행 관계 자동 파악
  - 병렬 처리 가능한 작업 식별
  - 크리티컬 패스 산출
- **MVP 범위**: 기본 순서 결정 (기획 → 디자인 → 개발 → QA 순)
- **수용 기준**: 의존성 없는 작업은 병렬 처리, 의존성 있는 작업은 순차 처리

---

### 3.3 Phase 3: 실행 시스템 (Execution)

#### FR-005: AI 기획 직원 자동 실행
- **설명**: 기획이 필요한 작업을 AI 기획 직원이 자동 수행
- **상세 요구사항**:
  - 요구사항을 구체적인 기획 문서(PRD, 기능 명세)로 변환
  - 사용자 스토리 자동 작성
  - 기획 결과를 다음 단계(디자인)에 전달
- **MVP 범위**: 간단한 기능 명세서 자동 생성

#### FR-006: AI 디자인 직원 자동 실행
- **설명**: 디자인이 필요한 작업을 AI 디자인 직원이 자동 수행
- **상세 요구사항**:
  - UI 변경 사항을 SwiftUI 코드로 반영할 수 있는 디자인 명세 생성
  - 색상, 레이아웃, 컴포넌트 변경 지시서 작성
  - 디자인 시스템 일관성 유지
- **MVP 범위**: SwiftUI 코드 레벨의 UI 변경 지시서 생성

#### FR-007: AI 개발 직원 자동 코드 생성
- **설명**: 개발 작업을 AI 개발 직원이 자동 수행
- **상세 요구사항**:
  - Claude Code 연동을 통한 코드 생성/수정
  - 기존 코드베이스 스타일 및 패턴 준수
  - 자동 코드 리뷰 (스스로 생성한 코드 검증)
  - Git 커밋 자동 생성
- **MVP 범위**: Claude Code CLI 호출을 통한 단일 파일 수정
- **수용 기준**: 생성된 코드가 빌드 에러 없이 컴파일되어야 함

#### FR-008: AI QA 직원 자동 테스트
- **설명**: 변경된 코드를 자동으로 테스트
- **상세 요구사항**:
  - 빌드 성공 여부 확인
  - 기존 테스트 통과 여부 확인
  - 새로운 기능에 대한 테스트 케이스 자동 생성
  - 실패 시 개발 직원에게 수정 요청 → 자동 수정 루프
- **MVP 범위**: 빌드 성공 여부 확인 + 에러 시 재시도 1회

---

### 3.4 Phase 4: 배포 시스템 (Deployment)

#### FR-009: 자동 빌드·배포 파이프라인
- **설명**: 테스트 통과 후 자동으로 빌드하고 배포
- **상세 요구사항**:
  - Tuist 기반 빌드 자동화
  - 코드 서명 자동 처리
  - App Store Connect 업로드 자동화
  - TestFlight 배포 자동화
- **MVP 범위**: 로컬 빌드 성공 확인까지만 (배포는 수동)

#### FR-010: 자동 수정 루프 (Self-Healing)
- **설명**: 빌드 실패 또는 테스트 실패 시 자동으로 원인을 분석하고 수정
- **상세 요구사항**:
  - 에러 로그 분석 → 원인 파악
  - 자동 수정 시도 (최대 3회)
  - 수정 불가 시 사용자에게 알림 + 원인 리포트
- **MVP 범위**: 빌드 에러 자동 수정 1회 시도

---

### 3.5 Phase 5: 피드백 & 관리 시스템

#### FR-011: 사용자 피드백 반영 사이클
- **설명**: 결과물에 대한 사용자 피드백을 받아 재실행
- **상세 요구사항**:
  - "여기 좀 더 수정해줘" 식의 추가 피드백 입력
  - 이전 작업 맥락을 유지한 채 수정
  - 변경 전후 비교(diff) 표시
- **MVP 범위**: 추가 텍스트 피드백 → 재실행

#### FR-012: 변경 이력 및 롤백
- **설명**: 모든 변경사항의 이력을 관리하고 필요 시 롤백
- **상세 요구사항**:
  - Git 기반 버전 관리
  - 특정 시점으로 롤백 가능
  - 변경 이력 타임라인 UI
- **MVP 범위**: Git 커밋 기반 롤백

#### FR-013: 22개 앱 멀티 프로젝트 관리
- **설명**: 여러 프로젝트를 통합 관리하는 대시보드
- **상세 요구사항**:
  - 프로젝트별 상태 한눈에 파악
  - 프로젝트 간 빠른 전환
  - 전체 프로젝트 일괄 업데이트 지원
- **MVP 범위**: 프로젝트 목록 + 기본 상태 표시

## 4. 기술 요구사항 (Non-Functional Requirements)

### NFR-001: 성능
- 요구사항 입력 → 작업 분해: 30초 이내
- 단순 코드 수정 (1파일): 5분 이내
- 복잡한 기능 추가 (다수 파일): 30분 이내

### NFR-002: 안정성
- 자동 수정 루프 최대 3회 → 실패 시 사용자 알림
- 작업 중 크래시 시 마지막 안정 상태로 자동 복구
- 모든 변경은 Git 커밋으로 보존

### NFR-003: 보안
- 사용자 API 키 안전 저장 (macOS Keychain)
- 코드 생성 시 보안 취약점 자동 검사
- 외부 전송 데이터 없음 (로컬 우선)

### NFR-004: 사용성
- 프로그래밍 지식이 없는 사람도 사용 가능
- 작업 진행 상황을 실시간으로 시각적 표시
- 각 AI 직원의 작업 과정을 투명하게 공개

## 6. 사용자 시나리오 (User Stories)

### US-001: 단순 UI 수정
> **사용자**: "메인 화면 배경색을 다크 그레이로 바꿔줘"
>
> **파이프라인 실행**:
> 1. 입력 수집 → 프로젝트 컨텍스트 로딩
> 2. 해석 → 디자인(색상 변경) + 개발(SwiftUI Color 수정)
> 3. 실행 → AI 개발 직원이 해당 파일의 `.background(Color.gray)` 수정
> 4. 검증 → 빌드 성공 확인
> 5. 완료 → 사용자에게 변경 사항 diff 표시
>
> **예상 소요**: 2~3분

### US-002: 새 기능 추가
> **사용자**: "설정 화면에 다크모드 토글 추가해줘"
>
> **파이프라인 실행**:
> 1. 입력 수집 → 설정 화면 코드 + 기존 테마 시스템 파악
> 2. 해석 → 기획(기능 명세) + 디자인(토글 UI) + 개발(상태 관리 + UI 구현)
> 3. 실행 → 순차적으로 명세 작성 → 코드 생성
> 4. 검증 → 빌드 + 기본 동작 확인
> 5. 완료 → 변경 파일 목록 + diff 표시
>
> **예상 소요**: 10~15분

### US-003: 버그 수정
> **사용자**: "로그인하면 앱이 크래시나"
>
> **파이프라인 실행**:
> 1. 입력 수집 → 로그인 관련 코드 + 크래시 로그 수집
> 2. 해석 → 개발(디버깅) + QA(재현 테스트)
> 3. 실행 → AI 개발 직원이 크래시 원인 분석 → 수정
> 4. 검증 → 빌드 + 크래시 재현 안 됨 확인
> 5. 완료 → 원인 분석 리포트 + 수정 내용 표시
>
> **예상 소요**: 5~10분

### US-004: 멀티 프로젝트 관리
> **사용자**: 22개 앱 중 5개에 대해 "iOS 18 호환성 업데이트해줘"
>
> **파이프라인 실행**:
> 1. 5개 프로젝트 각각의 컨텍스트 로딩
> 2. 각 프로젝트별 필요한 변경사항 분석
> 3. 병렬로 5개 프로젝트 수정 실행
> 4. 각각 빌드 확인
> 5. 전체 결과 대시보드에 표시
>
> **예상 소요**: 30분~1시간
