## 🎯 핵심 원칙

### 문서 기반 협업

```
❌ 잘못된 방식: 모든 내용을 대화로만 전달
   └─ 컨텍스트 손실, 부서 간 소통 단절

✅ 올바른 방식: 각 단계의 산출물을 문서로 저장
   └─ 다음 단계에서 해당 문서를 읽고 진행
   └─ 부서 간 명확한 인수인계
```

**가장 중요한 원칙**:
> 각 부서의 산출물은 반드시 **문서로 저장**하고,
> 다음 부서는 그 **문서를 읽고 작업**을 시작한다.

### 단계별 세션 분리

```
Phase 1: 기획팀 → 산출물 확인 → 수정 → 확정
         ↓ (문서 저장)
Phase 2: 디자인팀 → 산출물 확인 → 수정 → 확정
         ↓ (문서 저장)
Phase 3: 개발팀 → 구현 → 빌드 확인
         ↓ (코드 저장)
Phase 4: QA팀 → 테스트 → 버그 리포트
         ↓ (리포트 저장)
Phase 5: 마케팅팀 → 출시 준비
```

## 📁 프로젝트 구조

### 권장 디렉토리 구조

```
datas/[프로젝트명]/
├── _shared/
│   ├── documents/
│   │   ├── README.md           # 프로젝트 개요 (자동 생성)
│   │   ├── 01_PRD.md           # 기획 산출물
│   │   ├── 02_DESIGN_SPEC.md   # 디자인 스펙
│   │   └── 03_TASK_PLAN.md     # 개발 태스크
│   └── meetings/               # 회의록
│
├── 기획/
│   ├── documents/              # 기획서, 요구사항 정의서
│   ├── people/                 # 기획팀 직원 프로필
│   └── tasks/                  # 기획 태스크
│
├── 디자인/
│   ├── documents/              # 디자인 가이드, UI 명세서
│   │   └── mockups/            # HTML 목업 파일들
│   ├── people/
│   └── tasks/
│
├── 개발/
│   ├── documents/              # 기술 명세서, API 문서
│   ├── people/
│   └── tasks/
│
├── QA/
│   ├── documents/              # 테스트 계획서, 버그 리포트
│   ├── people/
│   └── tasks/
│
└── 마케팅/
    ├── documents/              # 마케팅 전략, 캠페인 기획
    ├── people/
    └── tasks/
```

### 문서 명명 규칙

| 유형 | 형식 | 예시 |
|------|------|------|
| 순서 문서 | `XX_문서명.md` | `01_PRD.md`, `02_DESIGN_SPEC.md` |
| 날짜 문서 | `YYYY-MM-DD-제목.md` | `2024-02-04-킥오프-회의록.md` |
| 버전 문서 | `제목-vX.X.md` | `API명세서-v1.2.md` |

## 👥 부서별 업무 가이드

### 1️⃣ 기획팀 (Planning)

**역할**: 프로젝트의 방향을 정의하고 요구사항을 문서화

**첫 대화 시작 방법**:
```
"[앱 이름]에 대한 PRD를 작성해줘.
타겟 사용자는 [대상]이고, 핵심 기능은 [기능1, 기능2, 기능3]이야."
```

**산출물 요청 형식**:
```
다음 포맷으로 PRD를 작성하고 문서로 저장해줘:

1. 제품 개요 (제품명, 한 줄 설명, 타겟 사용자, 핵심 가치)
2. 문제 정의 (사용자 문제 3가지, 기존 해결책의 한계)
3. 핵심 기능 (우선순위별 P0/P1/P2)
4. 화면 구성 (Screen Map)
5. 데이터 모델
6. 기술 요구사항
7. MVP 범위
8. 성공 지표 (KPI)
```

**확인 체크리스트**:
- [ ] 타겟 사용자가 명확한가?
- [ ] 핵심 기능의 우선순위가 정해졌는가?
- [ ] 화면 목록이 완전한가?
- [ ] MVP 범위가 현실적인가?

---

### 2️⃣ 디자인팀 (Design)

**역할**: PRD를 기반으로 UI/UX를 설계하고 시각화

**첫 대화 시작 방법**:
```
"기획팀이 작성한 PRD를 읽고 디자인 작업을 시작해줘.
datas/[프로젝트]/기획/documents/01_PRD.md 파일을 참고해."
```

**산출물 요청 형식**:
```
다음 산출물을 만들어줘:

1. 디자인 스펙 문서 (02_DESIGN_SPEC.md)
   - 컬러 팔레트 (HEX 값)
   - 타이포그래피 (사이즈/웨이트)
   - 스페이싱 체계 (4pt grid)
   - 컴포넌트 스타일

2. 각 화면별 HTML 목업
   - 모바일 뷰포트 (390x844)
   - iOS 스타일 UI
   - 인터랙티브 요소 포함
   - 화면 간 링크 연결
```

**디자인 레퍼런스 전달 방법**:
```
레퍼런스:
- 전체 톤: [Apple Health / Notion / 등]
- 컬러 무드: [미니멀 / 비비드 / 자연 친화적]
- 아이콘: SF Symbols [버전], [웨이트]
```

**확인 체크리스트**:
- [ ] 모든 화면이 목업으로 만들어졌는가?
- [ ] 디자인 시스템이 일관성 있는가?
- [ ] 다크모드가 고려되었는가?
- [ ] 접근성이 고려되었는가?

---

### 3️⃣ 개발팀 (Development)

**역할**: PRD와 디자인 스펙을 기반으로 실제 코드 구현

**첫 대화 시작 방법**:
```
"기획서와 디자인 스펙을 읽고 개발을 시작해줘.
- PRD: datas/[프로젝트]/기획/documents/01_PRD.md
- 디자인: datas/[프로젝트]/디자인/documents/02_DESIGN_SPEC.md
- 목업: datas/[프로젝트]/디자인/documents/mockups/"
```

**태스크 분해 요청**:
```
다음 순서로 태스크를 분해하고 03_TASK_PLAN.md에 저장해줘:

Layer 1: 기반 구조
- 프로젝트 생성
- 디자인 시스템 코드화
- 데이터 모델 정의
- 네비게이션 구조

Layer 2: 핵심 화면
- P0 기능 화면들
- HTML 목업과 1:1 일치하도록 구현

Layer 3: 비즈니스 로직
- ViewModel (TDD)
- 데이터 CRUD
- 외부 연동

Layer 4: 마감
- 에러 핸들링
- 로딩/빈 상태
- 다크모드
```

**구현 규칙**:
- 한 태스크 완료 시 빌드 확인
- 빌드 실패 시 즉시 수정
- 디자인 스펙의 값 정확히 적용
- 하드코딩 금지

---

### 4️⃣ QA팀 (Quality Assurance)

**역할**: 품질 보증 및 버그 탐지

**첫 대화 시작 방법**:
```
"PRD와 개발 완료된 코드를 기반으로 테스트 계획을 세워줘.
- PRD: datas/[프로젝트]/기획/documents/01_PRD.md
- 태스크: datas/[프로젝트]/개발/documents/03_TASK_PLAN.md"
```

**테스트 요청 형식**:
```
다음 테스트를 진행하고 QA 리포트를 작성해줘:

1. 기능 테스트 (PRD의 수용 기준 기반)
2. UI 테스트 (디자인 스펙과 일치 여부)
3. 엣지 케이스 테스트
4. 성능 테스트
5. 접근성 테스트
```

## 🔄 워크플로우 파이프라인

### 전체 흐름도

```
┌─────────────┐
│   기획팀    │ ──→ 01_PRD.md
└─────────────┘
       ↓ (문서 읽기)
┌─────────────┐
│  디자인팀   │ ──→ 02_DESIGN_SPEC.md + mockups/
└─────────────┘
       ↓ (문서 읽기)
┌─────────────┐
│   개발팀    │ ──→ 03_TASK_PLAN.md + 코드
└─────────────┘
       ↓ (코드 테스트)
┌─────────────┐
│    QA팀     │ ──→ 04_QA_REPORT.md
└─────────────┘
       ↓ (출시 준비)
┌─────────────┐
│  마케팅팀   │ ──→ 05_LAUNCH_PLAN.md
└─────────────┘
```

### 멘션(@)을 통한 협업

부서 간 협업이 필요할 때 멘션을 활용하세요:

```
@개발팀 이 기능의 기술적 구현 가능성을 검토해주세요.
@디자인팀 이 화면의 목업을 먼저 만들어주세요.
@QA팀 이 기능의 테스트 케이스를 작성해주세요.
```

### 문서화 요청

대화 내용을 문서로 정리하고 싶을 때:

```
"지금까지 논의한 내용을 문서화해줘"
"이 결정 사항을 위키에 정리해줘"
```

→ 자동으로 해당 부서의 documents 폴더에 저장됩니다.

## 💡 실전 예시

### 새 프로젝트 시작하기

```
1. 프로젝트 생성
   - 오피스 → 새 프로젝트 추가
   - 프로젝트명, 설명 입력

2. 기획팀 직원 배치
   - 프로젝트 오피스 → 기획팀 → 직원 추가
   - AI 유형: Claude 권장

3. 기획 시작
   - 기획팀 직원 클릭 → 대화 열기

   "반려동물 산책 기록 앱의 PRD를 작성해줘.

   핵심 기능:
   - 산책 경로 GPS 추적
   - 사진 기록
   - 건강 데이터 연동 (HealthKit)

   타겟: 반려견 보호자 (20-40대)

   PRD 작성 후 문서로 저장해줘."

4. 산출물 확인 및 수정
   - 회사 위키 또는 해당 부서 documents 폴더 확인
   - 수정 필요시 추가 대화

   "화면 구성에 '산책 통계 화면'을 추가해줘"
   "데이터 모델에 HealthKit 연동 필드 추가해줘"

5. 다음 단계로 이동
   - 디자인팀 직원 배치 → 대화 시작

   "기획팀이 작성한 PRD를 읽고 디자인 작업을 시작해줘.
   파일 위치: datas/[프로젝트]/기획/documents/01_PRD.md"
```

### 효율적인 대화 팁

**구체적으로 요청하기**:
```
❌ "앱 만들어줘"
✅ "산책 기록 앱의 홈 화면 PRD를 작성해줘.
   주요 기능은 오늘의 산책 요약, 빠른 산책 시작 버튼,
   최근 산책 기록 목록이야."
```

**레퍼런스 제공하기**:
```
"Apple Health 앱 스타일로 디자인해줘.
깔끔하고 여백이 충분한 느낌으로."
```

**단계별 확인하기**:
```
"먼저 화면 목록만 정리해줘" → 확인 →
"좋아, 이제 각 화면의 상세 스펙을 작성해줘"
```

## 🚀 고급 팁

### 1. 컨텍스트 관리

각 단계 완료 후 새 대화를 시작하면:
- 이전 대화의 컨텍스트 부담 없음
- 문서를 통해 필요한 정보만 전달
- 더 정확하고 집중된 결과물

```
Phase 1 완료 → "대화 초기화" 버튼 클릭 →
Phase 2 시작: "PRD 파일을 읽고 작업해줘"
```

### 2. 품질 높은 디자인 목업

디자인 요청 시 구체적 레퍼런스 추가:
```
디자인 레퍼런스:
- 전체 톤: Apple Health 앱 (깔끔, 여백 충분)
- 카드 UI: Notion 스타일 (라운드 코너, 미니멀 그림자)
- 컬러 무드: 자연 친화적 (그린 계열 Primary)
- 인터랙션: 부드러운 스프링 애니메이션
- 아이콘: SF Symbols 5.0, Regular 웨이트
```

### 3. 개발 시 목업 활용

HTML 목업을 SwiftUI로 변환할 때:
```
"이 HTML 목업 파일을 읽고 SwiftUI로 1:1 변환해줘.
컬러, 폰트 사이즈, 스페이싱, 코너 라디우스 모두 정확히 맞춰줘.
design/mockups/01_home.html → Views/HomeView.swift"
```

### 4. 협업 기록 활용

- 사이드바 → 협업 기록에서 부서 간 소통 내역 확인
- 이전 협업 내용을 참고하여 일관성 유지

### 5. 부서 스킬 커스터마이징

설정 → 부서 스킬에서:
- 각 부서의 역할 설명 수정
- 전문 분야 추가/변경
- 작업 스타일 조정
