# 픽셀 오피스 자동 개발 파이프라인 기획서 v1.0

> **핵심 명제**: 피드백/요구사항만 넣으면, AI가 알아서 기획→디자인→개발→테스트→배포하여 앱을 업데이트한다.

---

## 1. 왜 자동 개발 파이프라인인가

### 문제 정의
- 1인 창업자가 22개의 수익 앱을 유지·관리하는 것은 물리적으로 불가능
- 각 앱마다 기획→디자인→개발→QA→배포의 전체 프로세스를 매번 수동으로 수행해야 함
- 기존 도구(Cursor, Copilot 등)는 "코드 생성"만 도와줄 뿐, **개발 프로세스 전체를 자동화하지 못함**

### 해결 방향
- 사용자는 **"무엇을 원하는지"만 말하면 됨**
- AI 직원들이 체계적인 개발 프로세스를 거쳐 **실제 동작하는 코드를 만들고 배포**
- 사용자는 최종 결과를 확인하고 추가 피드백만 제공

## 2. 전체 파이프라인 구조

```
┌─────────────────────────────────────────────────────────────┐
│                    사용자 입력                                │
│  "로그인 화면에 소셜 로그인 추가해줘"                          │
│  "결제 버그 수정해줘" / "다크모드 넣어줘"                      │
└─────────────┬───────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  [1단계] 요구사항 해석 엔진                                   │
│  - 자연어 분석 → 구조화된 요구사항으로 변환                     │
│  - 기존 프로젝트 컨텍스트와 매핑                               │
│  - 영향 범위 분석 (어떤 파일, 어떤 기능에 영향?)                │
└─────────────┬───────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  [2단계] 작업 분해 & 할당                                     │
│  - 요구사항 → 기획/디자인/개발/QA 태스크 자동 분해              │
│  - 태스크 간 의존성 및 실행 순서 결정                           │
│  - AI 직원(에이전트)에게 태스크 자동 할당                       │
└─────────────┬───────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  [3단계] 자동 실행 파이프라인                                  │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ 기획 AI  │→│ 디자인 AI │→│ 개발 AI  │→│  QA AI   │    │
│  │ (PRD작성)│  │(UI명세)  │  │(코드생성)│  │(테스트)  │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
│                                                             │
│  각 단계의 산출물이 다음 단계의 입력이 됨                       │
└─────────────┬───────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  [4단계] 빌드 & 검증                                         │
│  - 자동 빌드 (Xcode/Tuist)                                   │
│  - 자동 테스트 실행                                           │
│  - 빌드 실패 시 → 자동 수정 루프 (Self-Healing)               │
└─────────────┬───────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  [5단계] 사용자 확인 & 배포                                   │
│  - 변경 사항 리포트 (무엇이 바뀌었는지 요약)                    │
│  - 사용자 승인 → App Store 배포                               │
│  - 또는 추가 피드백 → 1단계로 돌아감                           │
└─────────────────────────────────────────────────────────────┘
```

## 3. 각 단계 상세 설계 방향

### 3.1 요구사항 해석 엔진

| 항목 | 설명 |
|------|------|
| **입력** | 자연어 텍스트 (+ 선택적으로 스크린샷, 참고 URL) |
| **출력** | 구조화된 요구사항 (타입, 영향 범위, 우선순위) |
| **핵심 로직** | 프로젝트 컨텍스트(코드, 문서, 히스토리) 기반 해석 |

**요구사항 타입 분류**:
- `feature`: 새 기능 추가
- `bugfix`: 버그 수정
- `improvement`: 기존 기능 개선
- `design`: UI/UX 변경
- `refactor`: 코드 구조 개선

### 3.2 작업 분해 엔진

하나의 요구사항이 어떻게 분해되는지 예시:

```
입력: "로그인 화면에 Apple 로그인 추가해줘"

분해 결과:
├── [기획] Apple 로그인 요구사항 정의
│   └── 산출물: 요구사항 문서
├── [디자인] 로그인 화면 Apple 버튼 배치 설계
│   └── 산출물: UI 명세
├── [개발] Apple Sign-In API 연동
│   └── 산출물: AuthService 코드 수정
├── [개발] 로그인 화면 UI 수정
│   └── 산출물: LoginView.swift 수정
├── [QA] Apple 로그인 테스트 시나리오 실행
│   └── 산출물: 테스트 결과 리포트
└── [배포] 빌드 & 스토어 제출
    └── 산출물: 새 버전 빌드
```

### 3.3 AI 직원별 실행 능력

| AI 직원 | 입력 | 출력 | 핵심 도구 |
|---------|------|------|----------|
| 기획 AI | 사용자 요구사항 | PRD, 요구사항 정의서 | 프로젝트 컨텍스트, 유사 사례 분석 |
| 디자인 AI | PRD | UI 명세서, 레이아웃 정의 | 디자인 시스템, 컴포넌트 라이브러리 |
| 개발 AI | UI 명세 + PRD | 실제 코드 (Swift) | Claude Code, 코드베이스 인덱스 |
| QA AI | 코드 변경 내역 | 테스트 결과 리포트 | 빌드 시스템, 테스트 프레임워크 |
| 배포 AI | 테스트 통과 빌드 | 스토어 제출 | App Store Connect API, Fastlane |

### 3.4 자동 수정 루프 (Self-Healing)

```
빌드/테스트 실패
    │
    ▼
에러 로그 분석 (AI)
    │
    ▼
원인 파악 & 수정 코드 생성
    │
    ▼
재빌드 & 재테스트
    │
    ├── 성공 → 다음 단계로
    └── 실패 → 재시도 (최대 3회)
              │
              └── 3회 초과 → 사용자에게 에스컬레이션
                  "이 문제는 직접 확인이 필요합니다"
```

### 3.5 사용자 확인 포인트 (체크포인트)

완전 자동이 목표이지만, 안전장치로 사용자 확인 포인트를 둔다:

| 체크포인트 | 설명 | 기본값 |
|-----------|------|--------|
| 요구사항 해석 후 | "이렇게 이해했는데 맞나요?" | 자동 통과 |
| 코드 생성 후 | 변경 파일 diff 요약 | 자동 통과 |
| 빌드 성공 후 | 배포 전 최종 확인 | **사용자 승인 필요** |
| 배포 완료 후 | 결과 리포트 | 자동 알림 |

사용자 설정으로 모든 체크포인트를 자동 통과/수동 승인으로 조절 가능.

## 4. 프로젝트 컨텍스트 관리

### 4.1 AI가 프로젝트를 이해하는 방법

22개 앱을 관리하려면 AI가 각 프로젝트의 맥락을 정확히 이해해야 한다.

```
프로젝트 컨텍스트
├── 코드 인덱스 (파일 구조, 주요 클래스/함수)
├── 아키텍처 문서 (자동 생성/수동 보완)
├── 변경 이력 (git log + AI 요약)
├── 기술 스택 정보
├── 디자인 시스템 (색상, 폰트, 컴포넌트)
└── 비즈니스 로직 (도메인 규칙)
```

### 4.2 컨텍스트 효율성

- **Lazy Loading**: 필요한 컨텍스트만 로드
- **캐싱**: 자주 쓰는 컨텍스트 캐시
- **요약**: 전체 코드가 아닌 구조화된 요약 사용
- **증분 업데이트**: 변경된 부분만 컨텍스트 갱신

## 5. 차별점: 왜 픽셀 오피스인가

| | 기존 도구 (Cursor, Copilot 등) | 픽셀 오피스 |
|---|---|---|
| **범위** | 코드 생성/편집 | 기획→디자인→개발→QA→배포 전체 |
| **입력** | 코드 레벨 프롬프트 | 자연어 요구사항 |
| **관리 단위** | 파일/함수 | 프로젝트 전체 |
| **프로세스** | 없음 (사용자가 관리) | 체계적 개발 프로세스 자동화 |
| **팀 시뮬레이션** | 없음 | AI 직원 팀이 역할 분담 |
| **멀티 프로젝트** | 없음 | 22개 앱 동시 관리 |
| **배포** | 없음 | 빌드→테스트→스토어 배포 자동화 |

## 6. MVP 범위 제안

### Must Have (없으면 안 되는 것)
1. 자연어 요구사항 입력
2. 요구사항 → 태스크 자동 분해
3. Claude Code 기반 코드 생성
4. 자동 빌드 & 에러 수정 루프
5. 변경 사항 리포트

### Should Have (있으면 좋은 것)
6. 디자인 AI의 UI 명세 자동 생성
7. 자동 테스트 실행
8. 프로젝트 컨텍스트 자동 인덱싱

### Nice to Have (나중에)
9. App Store 자동 배포
10. 멀티 프로젝트 대시보드
11. 음성/스크린샷 입력

## 7. 리스크 및 고려사항

| 리스크 | 영향 | 대응 방안 |
|--------|------|----------|
| AI 코드 품질 불안정 | 빌드 실패, 버그 | Self-Healing 루프 + 에스컬레이션 |
| 컨텍스트 한계 (토큰 제한) | 대규모 프로젝트 이해 부족 | 구조화된 요약 + 필요시 로드 |
| 빌드/배포 자동화 복잡도 | 개발 기간 증가 | Fastlane 등 기존 도구 활용 |
| 22개 앱 동시 관리 | 리소스/비용 | 우선순위 기반 스케줄링 |
